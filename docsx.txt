SECTION 1: PROJECT OVERVIEW — THE BIG IDEA
================================================================================

1.1 WHAT IS Q-GUARDIAN?
-----------------------
Q-Guardian is a full-stack cybersecurity platform purpose-built for Punjab
National Bank (PNB) to address the emerging threat of quantum computing
against classical encryption. It is a "Quantum Transition Intelligence"
tool — meaning it helps the bank understand WHERE its current cryptography
is vulnerable, WHEN quantum computers will be able to break it, and HOW
to migrate to quantum-safe (Post-Quantum Cryptography / PQC) algorithms
before it's too late.

In simpler terms: Quantum computers are coming. When they arrive, every
RSA key, every ECDSA certificate, every non-PQC cipher suite used by the
bank today will be crackable. Q-Guardian is the early-warning system and
migration planner that ensures PNB is ready before that happens.

1.2 WHY DOES THIS MATTER? (THE PROBLEM STATEMENT)
--------------------------------------------------
Banks like PNB handle the most sensitive data on the planet — financial
transactions, KYC records, account credentials, internal audit data. All
of this is currently protected by classical encryption algorithms like
RSA-2048 and ECDSA-256. These algorithms are considered secure TODAY, but
they will be completely broken once a Cryptographically Relevant Quantum
Computer (CRQC) arrives. Estimates say this could happen in 5 to 15 years.

The critical threat is called "Harvest Now, Decrypt Later" (HNDL):
  - Nation-state adversaries are ALREADY recording encrypted bank traffic
    today.
  - They store this encrypted data in massive archives.
  - Once quantum computers arrive, they'll decrypt ALL of it retroactively.
  - This means data encrypted TODAY is already at risk if it has a long
    shelf life (e.g., KYC data valid for 10+ years).

Additionally, the Reserve Bank of India (RBI) has issued the Cybersecurity
Framework (CSF) 2.0 which mandates strong encryption controls. Banks must
demonstrate compliance with specific annexure controls around TLS versions,
key management, and forward secrecy.

1.3 WHAT IS OUR AIM?
---------------------
  1. DISCOVER all cryptographic assets across PNB's infrastructure
     (subdomains, TLS certificates, cipher suites, key sizes).
  2. SCORE each asset's quantum readiness using a proprietary metric
     called Q-TRI (Quantum Transition Readiness Index).
  3. PREDICT when each asset becomes vulnerable using the Mosca Countdown
     Framework (a real academic model).
  4. QUANTIFY the Harvest-Now-Decrypt-Later (HNDL) data exposure for
     assets lacking Perfect Forward Secrecy.
  5. MAP findings to RBI CSF 2.0 regulatory controls automatically.
  6. GENERATE migration playbooks with production-ready configurations
     for each vulnerable asset.
  7. EXPORT board-level PDF reports and machine-readable CBOM (Cryptographic
     Bill of Materials) in CycloneDX-compatible format.
  8. PROVIDE a built-in rule-based security chatbot for on-demand
     intelligence queries — without any external AI/model dependency.
  9. SCAN API endpoints against OWASP API Security Top 10.
  10. FEED live threat intelligence from quantum computing and
      cybersecurity RSS sources.

1.4 WHAT MAKES Q-GUARDIAN DIFFERENT?
-------------------------------------
  - NO EXTERNAL AI MODELS: The chatbot and all intelligence is rule-based
    and runs entirely locally. No data leaves the bank's network. This is
    critical for PNB's security posture.
  - REAL SCANNING: Uses actual TLS/SSL handshakes, DNS resolution, port
    scanning, and HTTP header analysis — not simulated data.
  - BANK-SPECIFIC BRANDING: The entire UI follows PNB's official brand
    guidelines (Maroon #A20C39, Gold #FBBC09) and includes PNB logos.
  - REGULATORY-FIRST: Every finding is automatically mapped to RBI CSF 2.0
    annexure controls with specific remediation guidance.
  - QUANTUM-FOCUSED: Built specifically for the quantum threat, not a
    generic vulnerability scanner.

3.1 main.py — THE API SERVER
-----------------------------
File: backend/app/main.py
Purpose: This is the heart of the backend. It defines ALL REST API endpoints
         and orchestrates every engine.

Key Design Decisions:
  - Uses FastAPI for automatic OpenAPI documentation, Pydantic validation,
    dependency injection, and async support.
  - CORS is set to allow all origins ("*") because the frontend runs on a
    different port during development (5173 vs 8000).
  - Background tasks are used for scanning so the user doesn't wait for the
    full scan to complete — results stream in via polling.
  - Database sessions are injected using FastAPI's Depends() mechanism to
    ensure proper session lifecycle management.

API Endpoints:
  GET  /                                → Health check
  POST /api/v1/scan/trigger             → Trigger a full domain scan
  POST /api/v1/scan/api                 → Trigger an API security scan
  GET  /api/v1/scan/{job_id}/status     → Poll scan progress (0-100%)
  GET  /api/v1/assets                   → List all discovered assets
  GET  /api/v1/enterprise/rating        → Get overall enterprise cyber rating
  GET  /api/v1/migration/{id}/playbook  → Get PQC migration playbook for asset
  GET  /api/v1/reports/board-brief      → Download PDF board brief
  GET  /api/v1/compliance/rbi           → Get RBI CSF 2.0 compliance mapping
  POST /api/v1/chat                     → Send message to chatbot
  GET  /api/v1/threat-intel             → Get live threat intelligence feed

Scan Workflow Logic:
  When the user clicks "TRIGGER FULL SCAN" on the frontend:
  1. Frontend POSTs to /scan/trigger with { domain: "example.com" }
  2. Backend creates a DBScanJob record with status "SCANNING"
  3. Backend spawns a background task (process_scan_background)
  4. Frontend immediately gets back a job_id
  5. Frontend polls /scan/{job_id}/status every 1 second
  6. Background task runs through these phases:
     a. 5%  → Initializing
     b. 10% → Running subdomain discovery (crt.sh + DNS + TLS)
     c. 30% → Discovery complete, starting deep scan per asset
     d. 30-70% → For each asset: port scan + Q-TRI + Mosca + HNDL
     e. 90% → Finalizing compliance mapping
     f. 100% → Scan Complete, status → "COMPLETED"
  7. Frontend detects COMPLETED, fetches all assets, shows Dashboard

Progress Tracking Logic:
  The update_job_progress() function is called throughout the scan to update
  the database with the current progress percentage and step description.
  This allows the frontend to show a live progress bar with messages like
  "Scanning api.pnb.bank.in ports... (45%)"

Enterprise Cyber Rating Logic:
  - Collects all Q-TRI scores from discovered assets
  - Calculates average Q-TRI score
  - Scales from 0-100 to 0-1000
  - Assigns a letter grade:
      > 700 → "A — Excellent"
      > 400 → "B/C — Good"
      > 200 → "D — Needs Work"
      else  → "F — Insecure"


3.2 database.py — DATA PERSISTENCE
------------------------------------
File: backend/app/database.py
Purpose: Defines the SQLite database schema and provides session management.

Why SQLite?
  - No need for a separate database server (PostgreSQL, MySQL, etc.)
  - The database is a single file (qguardian.db) — easy to backup and deploy
  - Perfectly adequate for a single-user security audit tool
  - check_same_thread=False is set because FastAPI runs async and background
    tasks may access the DB from different threads

Tables:

  DBScanJob:
    - id (int, primary key, auto-increment)
    - job_uuid (string, indexed) — UUID for each scan job
    - domain (string) — the domain that was scanned
    - status (string) — PENDING | SCANNING | COMPLETED | FAILED
    - progress (int) — 0 to 100 percentage
    - current_step (string) — human-readable step description
    - created_at (string) — ISO timestamp of when scan began
    - completed_at (string, nullable) — ISO timestamp of completion

  DBAsset:
    - id (int, primary key, auto-increment)
    - asset_uuid (string, indexed) — unique ID for each asset
    - job_uuid (string, indexed) — links asset to its parent scan job
    - hostname (string) — the discovered hostname
    - tls_version (string) — TLS version detected (1.0, 1.2, 1.3)
    - algorithm (string) — cryptographic algorithm (RSA-2048, ECDSA-256, etc.)
    - key_size (int) — key size in bits
    - cipher_suite (string) — full cipher suite string
    - forward_secrecy (bool) — whether PFS is enabled
    - cert_valid (bool) — whether the certificate is valid
    - cert_expiry (string) — certificate expiration date
    - sensitivity_tier (string) — S1 (most sensitive) to S5 (public)
    - is_pqc (bool) — whether post-quantum cryptography is in use
    - policy_compliant (bool) — whether asset meets TLS 1.3 policy
    - qtri_score (int) — calculated Q-TRI score (0-100)
    - mosca_data (string) — JSON-serialized Mosca countdown data
    - hndl_data (string, nullable) — JSON-serialized HNDL exposure data
    - last_scanned (string) — ISO timestamp of last scan


3.3 discovery.py — SUBDOMAIN DISCOVERY + TLS SCANNING ENGINE
--------------------------------------------------------------
File: backend/app/engines/discovery.py
Purpose: Discovers all subdomains of a domain and performs real TLS handshakes
         to extract cryptographic details from each.

Logic Breakdown:

  Step 1: Subdomain Discovery (get_subdomains_from_crtsh)
    - Queries crt.sh — a free Certificate Transparency (CT) log aggregator
    - URL: https://crt.sh/?q={domain}&output=json
    - CT logs are public records of every SSL certificate ever issued
    - This gives us a list of all subdomains that have certificates
    - Example: for "pnb.bank.in" we might discover:
        api.pnb.bank.in, auth.pnb.bank.in, vpn.pnb.bank.in, etc.

  Step 2: TLS Handshake (get_real_tls_info)
    - For each discovered hostname, we make a REAL SSL/TLS connection
    - We connect to port 443 with a 3-second timeout
    - We extract:
        * TLS version (1.0, 1.1, 1.2, 1.3)
        * Cipher suite name (e.g., TLS_AES_256_GCM_SHA384)
        * Whether Forward Secrecy is enabled (checks for EECDH/DHE)
        * Algorithm type (RSA vs ECDSA, derived from cipher suite)
        * Key size (2048 for RSA, 256 for ECDSA)
        * Whether PQC is in use (checks for KYBER/KEM in cipher name)
    - Certificate verification is disabled (CERT_NONE) so we can inspect
      even expired or self-signed certificates

  Step 3: Sensitivity Classification (scan_asset)
    - Each asset gets a sensitivity tier based on hostname patterns:
        "api" in hostname → S1 (Core Payment, 10-year shelf life)
        "auth" in hostname → S2 (KYC/Identity, 7-year shelf life)
        "vpn" in hostname → S3 (Transaction History, 5-year shelf life)
        default → S5 (Public Info, 1-year shelf life)
    - Policy compliance is determined by TLS version (must be 1.3)

  Step 4: Run Discovery (run_discovery)
    - Combines subdomain listing + TLS scanning
    - Limits to top 5 subdomains for speed during standard scans
    - Returns a list of asset dictionaries with all cryptographic metadata

Why This Matters:
  This is the foundation of everything. Without knowing WHAT infrastructure
  you have and WHAT cryptography it uses, you can't assess quantum risk.
  Certificate Transparency logs are a brilliant real-world data source that
  gives us genuine subdomain intelligence without needing internal network
  access.


3.4 port_scanner.py — ASYNCHRONOUS TCP PORT SCANNER
-----------------------------------------------------
File: backend/app/engines/port_scanner.py
Purpose: Scans each discovered host for open ports to identify exposed services.

Logic Breakdown:
  - Defines 17 common ports to scan:
      80 (HTTP), 443 (HTTPS), 21 (FTP), 22 (SSH), 23 (Telnet),
      25 (SMTP), 53 (DNS), 110 (POP3), 143 (IMAP), 465 (SMTPS),
      587 (Mail), 993 (IMAPS), 995 (POP3S), 3306 (MySQL),
      3389 (RDP), 5432 (PostgreSQL), 8080 (HTTP-Proxy), 8443 (HTTPS-Alt)

  - Uses Python's asyncio for concurrent scanning:
      * Each port is scanned simultaneously (not one at a time)
      * asyncio.open_connection() attempts a TCP handshake
      * 1-second timeout per port
      * If connection succeeds → port is OPEN
      * If timeout/refused/error → port is CLOSED

  - The run_port_scan() function is a synchronous wrapper that creates
    an asyncio event loop when needed (since the background scan task
    is synchronous)

Why This Matters:
  Open ports reveal the attack surface. An open MySQL port (3306) on a
  public-facing server is a critical misconfiguration. An open Telnet port
  (23) means unencrypted remote access. This data feeds into the asset
  inventory and helps security teams prioritize remediation. Port data
  is displayed in the Asset Inventory table on the frontend.


3.5 api_scanner.py — OWASP API SECURITY TOP 10 SCANNER
--------------------------------------------------------
File: backend/app/engines/api_scanner.py
Purpose: Tests an API endpoint against the OWASP API Security Top 10 checklist.

OWASP Checks Implemented:
  API1 — Broken Object Level Authorization (framework check)
  API2 — Broken Authentication (detected if no 401/403 on endpoints)
  API3 — Excessive Data Exposure (detected if Swagger/OpenAPI is public)
  API4 — Lack of Rate Limiting (detected by sending 15 rapid requests)
  API5 — Broken Function Level Authorization (framework check)
  API7 — Security Misconfiguration (detected via wildcard CORS)

Logic Breakdown:

  Step 1: Connectivity Test
    - Sends a GET request to the target URL
    - If connection fails, returns error immediately

  Step 2: Security Header Analysis (analyze_security_headers)
    - Checks for 7 critical HTTP security headers:
        * HSTS (Strict-Transport-Security) — prevents downgrade attacks
        * CSP (Content-Security-Policy) — prevents XSS
        * X-Frame-Options — prevents clickjacking
        * X-Content-Type-Options — prevents MIME sniffing
        * X-XSS-Protection — legacy XSS protection
        * Cache-Control (no-store) — prevents caching sensitive data
        * CORS (Access-Control-Allow-Origin) — cross-origin policy
    - Calculates a header score: (headers present / 7) * 100

  Step 3: Rate Limiting Test (API4)
    - Sends 15 rapid GET requests to the target
    - If no HTTP 429 (Too Many Requests) is ever returned, the API
      lacks rate limiting → HIGH severity finding

  Step 4: CORS Policy Check (API7)
    - If Access-Control-Allow-Origin is "*" (wildcard), it means any
      website can make requests to this API → MEDIUM severity finding

  Step 5: Endpoint Discovery & Auth Check (API2, API3)
    - Probes common API paths: /api, /v1, /api/v1, /swagger.json,
      /openapi.json, /health
    - If ALL return 200 (no 401/403), it means NO authentication
      is required → CRITICAL severity finding
    - If /swagger.json or /openapi.json is publicly accessible,
      API documentation is exposed → MEDIUM severity (API3)

  Step 6: Information Disclosure
    - Checks if Server header reveals server software (e.g., "nginx/1.20")
    - Checks if X-Powered-By reveals framework (e.g., "Express")

  Step 7: Risk Score Calculation
    - Starts with header_score
    - Subtracts 10 points per OWASP finding
    - Clamped between 0 and 100

Why This Matters:
  APIs are the primary attack vector for modern banking systems. Every
  mobile banking app, UPI integration, and partner system connects via
  APIs. Testing them against OWASP's API-specific security standard
  is critical for PNB's security posture.


3.6 scoring.py — Q-TRI (QUANTUM TRANSITION READINESS INDEX)
-------------------------------------------------------------
File: backend/app/engines/scoring.py
Purpose: Calculates a proprietary 0-100 score for each asset's quantum readiness.

Q-TRI Score Components:
  ┌─────────────────────────────┬────────┬──────────────────────────┐
  │ Component                   │ Weight │ How It's Scored          │
  ├─────────────────────────────┼────────┼──────────────────────────┤
  │ PQC Adoption                │  30%   │ 30 if ML-KEM/ML-DSA used │
  │ TLS Version                 │  20%   │ 20 for 1.3, 10 for 1.2  │
  │ Forward Secrecy             │  15%   │ 15 if ECDHE/DHE enabled  │
  │ Certificate Hygiene         │  15%   │ 15 if valid + key≥2048   │
  │ Cipher Suite Strength       │  10%   │ 10 if GCM or CHACHA20    │
  │ Policy Compliance           │  10%   │ 10 if passes all checks  │
  └─────────────────────────────┴────────┴──────────────────────────┘

  Maximum score: 100 (fully quantum-safe)
  Minimum score: 0 (completely vulnerable)

Enterprise Cyber Rating:
  - Averages all asset Q-TRI scores
  - Scales to 0-1000 range (avg * 10)
  - This gives a single number for the board to track progress

Why This Matters:
  The Q-TRI score is the central metric of the entire platform. It gives
  a single, comparable number that answers "How quantum-ready is this
  asset?" Without this, security teams would need to manually evaluate
  dozens of cryptographic parameters per asset. The weighting reflects
  real-world priorities — PQC adoption (30%) is weighted highest because
  it's the most important defense against quantum threats.


3.7 mosca.py — MOSCA COUNTDOWN CLOCK
--------------------------------------
File: backend/app/engines/mosca.py
Purpose: Implements the Mosca Risk Framework to calculate when assets become
         vulnerable to quantum attack.

The Mosca Inequality (Academic Model):
  X + Y > Z means the asset IS at risk.

  Where:
    X = Time needed to migrate to PQC (in years)
        Based on migration_complexity (0.5 to 3.0 years)
    Y = Data shelf life (how long the data must remain secret)
        S1: 10 years (core payment systems)
        S2: 7 years (KYC/identity data)
        S3: 5 years (transaction history)
        S4: 3 years (internal business)
        S5: 1 year (public information)
    Z = Time until CRQC arrives
        Worst case: 5 years
        Best case: 15 years

  Calculation:
    days_remaining = (Z - (X + Y)) * 365.25

  Risk Classification:
    CRITICAL → days_remaining (worst case) ≤ 0
                (Migration deadline has ALREADY passed)
    WARNING  → days_remaining (worst case) < 365
                (Less than 1 year to act)
    MONITOR  → days_remaining (best case) < 3 years
                (Should start planning)
    SAFE     → Plenty of time remaining

Example:
  Asset: api.pnb.bank.in (S1 tier)
    X = 1.75 years (moderate complexity)
    Y = 10 years (core payment data shelf life)
    Z = 5 years (worst case CRQC)
    X + Y = 11.75 years > Z = 5 years
    days_remaining = (5 - 11.75) * 365.25 = -2467 days
    STATUS: CRITICAL (deadline passed 6.75 years ago!)

Why This Matters:
  Mosca is not our invention — it's a widely accepted academic framework
  by Dr. Michele Mosca for evaluating quantum risk timelines. By
  implementing it computationally, we give PNB a scientific, defensible
  basis for prioritizing migration efforts. The "countdown clock" concept
  is powerful for communicating urgency to non-technical executives.


3.8 hndl.py — HARVEST-NOW-DECRYPT-LATER EXPOSURE CALCULATOR
--------------------------------------------------------------
File: backend/app/engines/hndl.py
Purpose: Quantifies the data volume at risk from HNDL attacks.

HNDL Only Applies To:
  Assets WITHOUT Perfect Forward Secrecy (PFS). If an asset uses PFS
  (ECDHE/DHE key exchange), each session has unique keys, so even if
  the long-term private key is compromised, past sessions cannot be
  decrypted. Without PFS, ALL historical traffic can be decrypted once
  the key is broken.

Logic:
  - Traffic volume (GB/month) is estimated based on sensitivity tier:
      S1: 500 GB/month (core payment APIs handle massive traffic)
      S2: 300 GB/month (authentication services)
      S3: 150 GB/month (transaction logs)
      S4: 50 GB/month (internal systems)
      S5: 10 GB/month (public-facing pages)
  - A deterministic factor based on hostname length adds variation
  - Harvest start date is assumed to be January 1, 2023
  - Months captured = (today - Jan 2023) / 30
  - total_gb_at_risk = traffic_volume * months_captured
  - Risk value = total_gb_at_risk * sensitivity_multiplier
      (S1: ×10, S2: ×5, S3: ×2, S4: ×1, S5: ×0.5)

Why This Matters:
  HNDL is the most immediate quantum threat because it's happening RIGHT
  NOW. Adversaries don't need quantum computers to harvest data — they
  just need to record encrypted traffic. By quantifying the data volume
  at risk, we help PNB understand the scale of potential exposure and
  justify the urgency of enabling PFS everywhere.


3.9 cbom.py — CRYPTOGRAPHIC BILL OF MATERIALS GENERATOR
---------------------------------------------------------
File: backend/app/engines/cbom.py
Purpose: Generates a machine-readable inventory of all cryptographic assets
         in CycloneDX-compatible JSON format.

CBOM Structure:
  {
    "metadata": {
      "timestamp": "2026-04-11T...",
      "tool": "Q-Guardian v2.0",
      "format": "CycloneDX-Compatible"
    },
    "components": [
      {
        "name": "api.pnb.bank.in",
        "type": "service",
        "cryptography": {
          "algorithm": "RSA-2048",
          "key_size": 2048,
          "tls_version": "1.2",
          "forward_secrecy": false
        },
        "risk": {
          "qtri_score": 25,
          "mosca_status": "CRITICAL"
        }
      }
    ]
  }

Why This Matters:

3.10 migration.py — PQC MIGRATION PLAYBOOK GENERATOR
------------------------------------------------------
File: backend/app/engines/migration.py

  RSA-based assets:

    Config: SSL protocols set to TLS 1.3

    Target: Hybrid ECC + ML-KEM-512
    Config: Key exchange curve: SecP256r1MLKEM512:prime256v1
    Effort: 1 to 3 weeks

    Marked as URGENT with 1-week protocol upgrade requirement

    - Current cryptographic state
    - Target PQC algorithm with NIST FIPS reference

    - Effort estimates
    - Risk reduction impact description

Why This Matters:
  It's not enough to tell a security team "you're vulnerable." They need
  to know EXACTLY what to do. The playbook gives them production-ready
  Nginx configurations they can copy-paste into their server configs.
  The hybrid approach (RSA + ML-KEM) is the NIST-recommended transition
  strategy — it maintains backward compatibility while adding quantum
  resistance.

SECTION 5: DATABASE DESIGN AND DATA FLOW
================================================================================

5.1 DATA LIFECYCLE
-------------------
  1. User enters domain → clicks "TRIGGER FULL SCAN"
  2. DBScanJob created with status=SCANNING, progress=0
  3. Background task discovers subdomains via crt.sh
  4. For each subdomain:
     a. Real TLS handshake → extract crypto metadata
     b. Async port scan → identify open services
     c. Calculate Q-TRI score
     d. Calculate Mosca countdown
     e. Calculate HNDL exposure (if no PFS)
     f. Create DBAsset record, commit to database
     g. Update DBScanJob progress
  5. Scan completes → DBScanJob status=COMPLETED
  6. Frontend fetches all DBAssets via /api/v1/assets
  7. Enterprise rating calculated on-the-fly from all Q-TRI scores
  8. Dashboard, charts, and tables render from this data

5.2 WHY WE COMMIT EACH ASSET INDIVIDUALLY
-------------------------------------------
In main.py, each asset is committed to the database immediately after
processing (session.commit() inside the loop). This is intentional:
  - Assets appear in the database DURING the scan, not only at the end
  - If the scan fails midway, already-processed assets are preserved
  - The frontend could theoretically show partial results during scanning
  - Fresh sessions per asset avoid SQLite lock contention

SECTION 6: SECURITY CONSIDERATIONS
================================================================================

6.1 NO EXTERNAL AI DEPENDENCY
-------------------------------
The chatbot uses a local rule-based engine. No user data is sent to
OpenAI, Google, or any external LLM service. This is by design — a
banking cybersecurity tool must not leak internal questions or scan
data to third parties.

6.2 CORS POLICY
-----------------
Currently set to allow_origins=["*"] for development. In production,
this MUST be restricted to the frontend's specific origin.

6.3 TLS SCANNING APPROACH
---------------------------
When scanning TLS, we disable certificate verification (CERT_NONE).
This is intentional — we NEED to inspect even expired or self-signed
certificates. We're auditing, not trusting.

6.4 DATA RESIDENCY
--------------------
All data stays in the local SQLite database. No cloud services are
used for data storage.

6.5 SSL WARNINGS SUPPRESSED
-----------------------------
In api_scanner.py, urllib3 InsecureRequestWarning is disabled because
we intentionally make unverified HTTPS requests during scanning.

SECTION 8: KEY ALGORITHMS AND FORMULAS
================================================================================

8.1 Q-TRI SCORE
-----------------
  QTRI = (PQC * 30) + (TLS * 20) + (PFS * 15) + (CertHealth * 15)
         + (CipherStrength * 10) + (PolicyCompliance * 10)

  Where each component is binary (0 or its weight).
  Maximum: 100 | Minimum: 0

8.2 MOSCA COUNTDOWN
---------------------
  Risk exists when: X + Y > Z

  days_remaining = (Z - (X + Y)) * 365.25

  X = 0.5 + (migration_complexity * 2.5)  [0.5 to 3.0 years]
  Y = SENSITIVITY_SHELF_LIFE[tier]         [1 to 10 years]
  Z = CRQC_TIMELINE                        [5 to 15 years]

8.3 HNDL RISK SCORE
---------------------
  traffic_volume = base_volume[tier] + (hostname_length * 5)
  months_captured = (today - Jan 1, 2023) / 30
  total_gb_at_risk = traffic_volume * months_captured
  hndl_risk_score = total_gb_at_risk * sensitivity_multiplier[tier]

8.4 ENTERPRISE CYBER RATING
------------------------------
  rating = (average of all Q-TRI scores) * 10
  Scale: 0 to 1000

8.5 API RISK SCORE
--------------------
  api_risk_score = header_score - (owasp_findings_count * 10)
  Clamped to [0, 100]

  header_score = (security_headers_present / 7) * 100

public sector bank operating under strict RBI regulatory requirements.

Every feature has a clear purpose:
  - Discovery Engine answers: "WHAT do we have?"
  - Q-TRI Scoring answers: "HOW ready is each asset?"
  - Mosca Countdown answers: "WHEN does each asset become vulnerable?"
  - HNDL Simulator answers: "HOW MUCH data is at risk RIGHT NOW?"
  - API Scanner answers: "Are our APIs secured against known attacks?"
  - Compliance Mapper answers: "Are we meeting RBI requirements?"
  - Migration Playbooks answer: "WHAT EXACTLY should we change?"
  - Board Brief answers: "How do we communicate this to leadership?"
  - CBOM Export answers: "How do we share this with auditors?"
  - Chatbot answers: "Can our team get instant answers?"
  - Threat Intel answers: "What's happening in the quantum world?"

The entire system runs locally, uses no external AI, stores data in a
single SQLite file, and follows PNB's exact brand guidelines. It is
designed to be deployed, demonstrated, and used as a production-grade
tool for India's transition to quantum-safe banking infrastructure.
